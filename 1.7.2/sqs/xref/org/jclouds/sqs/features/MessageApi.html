72      * The DeleteMessage action deletes the specified message from the specified
73      * queue. You specify the message by using the message's receipt handle and
74      * not the message ID you received when you sent the message. Even if the
75      * message is locked by another reader due to the visibility timeout setting,
76      * it is still deleted from the queue. If you leave a message in the queue
77      * for more than 4 days, SQS automatically deletes it.
78      * 
79      * <h4>Note</h4>
80      * 
81      * The receipt handle is associated with a specific instance of receiving the
82      * message. If you receive a message more than once, the receipt handle you
83      * get each time you receive the message is different. When you request
84      * DeleteMessage, if you don't provide the most recently received receipt
85      * handle for the message, the request will still succeed, but the message
86      * might not be deleted.
87      * 
88      * <h4>Important</h4>
89      * 
90      * It is possible you will receive a message even after you have deleted it.
91      * This might happen on rare occasions if one of the servers storing a copy
92      * of the message is unavailable when you request to delete the message. The
93      * copy remains on the server and might be returned to you again on a
94      * subsequent receive request. You should create your system to be idempotent
95      * so that receiving a particular message more than once is not a problem.
96      * 
97      * @param queue
98      *           the queue the message is in
99      * @param receiptHandle
100     *           The receipt handle associated with the message you want to
101     *           delete.
102     */
103    @Named("DeleteMessage")

156    /**
157     * The ChangeMessageVisibility action changes the visibility timeout of a
158     * specified message in a queue to a new value. The maximum allowed timeout
159     * value you can set the value to is 12 hours. This means you can't extend
160     * the timeout of a message in an existing queue to more than a total
161     * visibility timeout of 12 hours. (For more information visibility timeout,
162     * see Visibility Timeout in the Amazon SQS Developer Guide.)
163     * 
164     * For example, let's say the timeout for the queue is 30 seconds, and you
165     * receive a message. Once you're 20 seconds into the timeout for that
166     * message (i.e., you have 10 seconds left), you extend it by 60 seconds by
167     * calling ChangeMessageVisibility with VisibilityTimeoutset to 60 seconds.
168     * You have then changed the remaining visibility timeout from 10 seconds to
169     * 60 seconds.
170     * 
171     * <h4>Important</h4>
172     * 
173     * If you attempt to set the VisibilityTimeout to an amount more than the
174     * maximum time left, Amazon SQS returns an error. It will not automatically
175     * recalculate and increase the timeout to the maximum time remaining.
176     * 
177     * <h4>Important</h4>
178     * 
179     * Unlike with a queue, when you change the visibility timeout for a specific
180     * message, that timeout value is applied immediately but is not saved in
181     * memory for that message. If you don't delete a message after it is
182     * received, the visibility timeout for the message the next time it is
183     * received reverts to the original timeout value, not the value you set with
184     * the ChangeMessageVisibility action.
185     * 
186     * @param queue
187     *           the queue the message is in
188     * @param receiptHandle
189     *           The receipt handle associated with the message whose visibility
190     *           timeout you want to change. This parameter is returned by the
191     *           ReceiveMessage action.
192     * @param visibilityTimeout
193     *           The new value for the message's visibility timeout (in seconds)
194     *           from 0 to 43200 (maximum 12 hours)
195     */
